# Лабораторная работа №4 – Интерфейсы и абстрактные классы (ABC)

## Цель работы
- Познакомиться с абстрактными базовыми классами (ABC) в Python.
- Освоить понятие интерфейса (контракта поведения).
- Научиться задавать обязательные методы для классов.
- Закрепить полиморфизм через единый интерфейс.
- Научиться проектировать архитектуру, а не просто классы.

## Описание интерфейсов

Создано два интерфейса (абстрактных класса) в файле `interfaces.py`:

| Интерфейс     | Абстрактный метод | Назначение |
|---------------|-------------------|-------------|
| `Printable`   | `to_string() -> str` | Возвращает строковое представление объекта |
| `Comparable`  | `compare_to(other) -> int` | Сравнивает текущий объект с другим. Возвращает -1 (меньше), 0 (равно), 1 (больше) |

Оба интерфейса наследуют от `ABC` и используют декоратор `@abstractmethod`.

## Реализация в классах

Интерфейсы реализованы в классах банковских счетов:

| Класс           | Наследует интерфейсы | Поведение `to_string()` | Поведение `compare_to(other)` |
|----------------|----------------------|--------------------------|-------------------------------|
| `CreditAccount` | `Printable`, `Comparable` | Возвращает строку вида: `"Кредитный счёт: владелец, номер, баланс, валюта, активен, ставка=...%"` | Сравнение **по значению баланса** (отрицательный баланс считается меньше положительного) |
| `SaveAccount`   | `Printable`, `Comparable` | Возвращает строку вида: `"Сберегательный счёт: владелец, номер, баланс, валюта, активен, ставка=...%"` | Сравнение **по значению баланса** |

Метод `to_string()` в каждом классе использует уже существующий `__str__` для единообразия.  
Метод `compare_to()` реализован в базовом классе `BankAccount` и сравнивает счета по их денежному балансу.

## Демонстрация (сценарии из demo.py)

В файле `demo.py` реализовано 5 сценариев, демонстрирующих работу интерфейсов.

### Сценарий 1. Использование интерфейса как типа
Универсальная функция `print_all(items: list[Printable])` принимает список объектов, поддерживающих `Printable`, и вызывает для каждого `to_string()`.  
Показывает, что книгам и фильмам (а в нашем случае – счетам) не нужно знать друг о друге, достаточно общего контракта.

### Сценарий 2. Проверка `isinstance`
Для каждого объекта из коллекции выводится, реализует ли он интерфейсы `Printable` и `Comparable`. Это подтверждает, что все счета корректно наследуют интерфейсы.

### Сценарий 3. Фильтрация коллекции по интерфейсу
Коллекция `BankAccountCollection` имеет методы `get_printable()` и `get_comparable()`, которые возвращают только объекты, поддерживающие соответствующий интерфейс. В демо показано, что все счета попадают в оба списка (так как все реализуют оба интерфейса).

### Сценарий 4. Сравнение через `Comparable`
Функция `find_min(comparable_list)` находит объект с минимальным значением, используя метод `compare_to()`. Это демонстрирует, что можно полиморфно сравнивать любые объекты, реализующие `Comparable`, без приведения типов.

### Сценарий 5. Полиморфизм без `isinstance`
Простой цикл по всем объектам коллекции вызывает метод `to_string()` напрямую. **Никаких проверок `isinstance` перед вызовом** – это пример хорошего стиля: интерфейс гарантирует наличие метода.
